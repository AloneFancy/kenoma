How To Use Each Layer
1
CLAUDE.md — auto-loaded. Every Claude Code session in this folder starts with the full project context: game identity, system build order, Iron Laws, character rules, and command index. You never need to re-explain what KENOMA is.
2
Commands — invoke with /iron-check, /new-puzzle, etc. Each command is a structured prompt that enforces a specific workflow. They gate dangerous operations (like making puzzles without authors) and produce consistent output formats that feed back into the codebase.
3
Agents — invoked by telling Claude Code to "use the narrative-guardian agent" or "run the playtester on this scene." Each agent loads specific skills and operates with defined permissions. The narrative-guardian runs read-only; the unity-systems agent has write access.
4
Skills — loaded automatically by agents that list them. kenoma-core is the compact project context (game identity, Iron Laws, characters, worlds). unity-kenoma is the Unity-specific architecture. vanguard-voice is the dialogue writing guide. Skills reduce token cost by keeping agent context focused.
5
Rules — applied to every session. 10 inviolable rules derived from the Iron Laws and design principles. When Claude Code is about to do something that violates a rule, the rule stops it before it happens.

Read-only gatekeeper. Reviews any file against the five core ideas, Vanguard character consistency, and all four Iron Laws. Run before merging anything that touches Vanguard behavior, puzzle design, or BMS warmth logic. Produces a compliance report with PASS/FAIL per criterion.

Specialist for the Behavioral Model System — the spine of everything. Designs new event types, reviews warmth register logic, diagnoses surfacing failures. Knows the full 12 event type schema, both warmth registers, and why USS must be built last. Has write access to Scripts/BMS/ and Scripts/Vanguard/.

Agent Unity Systems use the unity-systems agent
Implements all 8 core systems in Unity 6 URP. Knows the build order (BMS first, USS last), system contracts, FMOD patterns, and project coding conventions. Will refuse to build USS before RSS/TDS/TVS are stable. Enforces all 10 project rules during implementation.

Agent Playtester use the playtester agent
Simulates a fresh player's first experience through a scene or puzzle. Writes a first-person session report describing what they understood, what confused them, and how the Vanguard seemed. Flags design failures: [BMS_NOT_SURFACING], [MECHANIC_VISIBLE], [WARMTH_DELIVERED_NOT_EARNED], etc. Read-only.

10 Project Rules · Applied Every Session
#RuleWhat It Prevents
1BMS data is never exposed to the playerWarmth registers visible in UI, logs, or player-facing feedback
2Never use inline strings for BMS tagsMagic strings in RecordEvent() calls
3No BMS events in Update loopsPer-frame event dispatch, continuous warmth accumulation
4The Iron Rule has no exceptionsCrossing logic that deviates — even once, even with good intent
5Vanguard adapts through prediction, not mirroringHello Neighbor failure mode: AI blocks last route instead of predicting next
6Every puzzle must have an author (4-field comment)Puzzles that exist because "this room needed a puzzle"
7Dark Voice is binaural from day oneReverb-based Dark Voice added in post-production
8All designer-tunable values are SerializeFieldHardcoded gameplay constants
9Debug code is always gated behind debugMode flagDebug.Log in production builds, warmth values in console
10Story authority over system authoritySystem implementation contradicting the story hierarchy
Recommended Workflow Per Sprint
①
Planning: /sprint-plan [system] → check build order → get day-by-day tasks with acceptance criteria
②
Before design: playtester agent on the intended experience → catch design failures before building. For puzzle design: /new-puzzle enforces Iron Law IV authorship before any code.
③
During implementation: unity-systems agent or bms-architect agent for system work. /bms-event for any new Scholar behavior. /crossing-logic for new TFPs.
④
Dialogue & voice: /vanguard-dialogue with Vanguard + stage + context. /dark-voice with scene moment + what to surface. Both commands enforce character register.
⑤
Before merge to dev: /iron-check on all new files. /scene-review on any changed scene. narrative-guardian agent on any Vanguard behavior or puzzle design change.
⑥
Before external showing: /demo-check — the single gate. One verdict only. If not externally showable, one blocking issue. Fix that. Run again.
The only verdict that matters for the demo: "The Vanguard treated me differently based on what I chose." If a playtester describes only their own actions and not the Vanguard's response — the BMS is running but not surfacing. That is a failure state.
